    # Atomic原子类

    # JUC包的4类原子类：

    • 基本类型

      • AtomicInteger
      • AtomicLong
      • AtomicBoolean

    • 数组类型

      • AtomicIntegerArray
      • AtomicLongArray
      • AtomicReferenceArray：引用类型数组原子类

    • 引用类型

      • AtomicReference
      • AtomicStampedReference：原子更新带有版本号的引用类型
      • AtomicMarkableReference：原子更新带标记位的引用类型

    • 对象的属性修改类型

      • AtomicIntegerFieldUpdater：原子更新整型字段的更新器
      • AtomicLongFieldUpdater：原子更新长整型字段的更新器
      • AtomicReferenceFieldUpdater：原子更新引用类型字段的更新器

    # 实现原理

    AtomicInteger 类主要利用 CAS (compare and swap) + volatile 和 native 方法来保证原子操作，从而避免 synchronized 的高开销，执行效率大为提升。

    # LockSupport详解

    # 核心函数

    • pack(boolean isAbsolute, long time)
    • unpack(Thread thread)

    # Thread.sleep()

    传入休眠时间，到时间后自动唤醒，期间不会释放锁

    # Object.wait()

    • 不带时间参数：需要Object.notity()手动唤醒
    • 带时间参数：期间没有用Object.notity唤醒，则时间到了会自动唤醒；
      • 唤醒后立即获得锁
      • 唤醒后拿不到锁，则加入同步等待队列

    # Condition.await()

    与Object.wait()原理基本一致，不过wait底层由LockSupport.park()来实现阻塞线程

    # Thread.sleep与LockSupport.park区别

    • 两者阻塞线程，都不会释放锁
    • Thread.sleep无法手动唤醒，时间到了自动唤醒；LockSupport.park可以通过LockSupport.unpark唤醒
    • Thread.sleep向上抛异常；LockSupport.park不会抛异常
    • Thread.sleep本身实现就是通过底层native实现；LockSupport.park是通过Unsafe实现，Unsafe通过底层native实现

    # Object.wait与LockSupport.park区别

    • Object.wait只能在sychronized代码块中执行；LockSupport.park可以在任意位置
    • Object.wait向上抛出异常，LockSupport.park不需要抛异常
    • Object.wait不带超时的方法，需要调用Object.notity唤醒，后续不一定继续执行线程的内容；LockSupport.park不带超时的方法，需要调用LockSupport.unpark方法唤醒，后续一定继续执行线程的内容

    # AQS

    # 核心思想

    当被请求的共享资源空闲中，那么当前线程设置为有效线程，同时将资源设置为锁定状态，这时其他线程过来请求共享资源时，发现资源被占用了，那么请求就会被放到CLH队列中进行阻塞，后续通过唤醒机制来唤醒分配请求。

    AQS通过int成员变量来表示同步状态，通过内置的 FIFO 队列来完成获取资源线程的排队工作。AQS 使用 CAS 对该同步状态进行原子操作实现对其值的修改。

//原子地（CAS操作）将同步状态值设置为给定值update如果当前同步状态的值等于expect（期望值）
protected final boolean compareAndSetState(int expect, int update) {
    return unsafe.compareAndSwapInt(this, stateOffset, expect, update);

    # AQS对资源的共享方式

    • Exclusive（独占）
    • Share（共享）

    自定义同步器在实现时只需要实现共享资源 state 的获取与释放方式即可，至于具体线程等待队列的维护（如获取资源失败入队/唤醒出队等），AQS 已经在顶层实现好了。

    # AQS底层使用了模板方法模式

    AQS 使用了模板方法模式，自定义同步器时需要重写下面几个 AQS 提供的钩子方法：

    protected boolean tryAcquire(int)//独占方式。尝试获取资源，成功则返回true，失败则返回false。
protected boolean tryRelease(int)//独占方式。尝试释放资源，成功则返回true，失败则返回false。
protected int tryAcquireShared(int)//共享方式。尝试获取资源。负数表示失败；0表示成功，但没有剩余可用资源；正数表示成功，且有剩余资源。
protected boolean tryReleaseShared(int)//共享方式。尝试释放资源，成功则返回true，失败则返回false。
protected boolean isHeldExclusively()//该线程是否正在独占资源。只有用到condition才需要去实现它。

    # Questions

    # 1.为什么AQS底层使用虚拟双向队列

    prev和next指针主要是**中断和唤醒后续阻塞线程**时需要用到

    • 中断：用于在**AQS#acquireInterruptibly(int arg)获取锁的过程中处理中断信号，其中调用了cancelAcquire(Node node)负责将被中断的线程节点从AQS同步队列中移除**。单向链表从中间删除一个节点效率较低。
    • 唤醒：在CLH的prev基础上增加了next。AQS维护了next指针，以便活跃线程释放锁后主动唤醒后续阻塞线程去竞争锁
