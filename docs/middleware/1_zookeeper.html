    # 什么是zookeeper

    zookeeper是一款为分布式架构提供服务的一款高可用软件，它为分布式计算提供了分布式配置、同步和注册服务；分布式应用可以基于它实现数据发布/订阅、负载均衡、命名服务、集群管理、master选举、分布式锁、分布式队列等功能。

    # zookeeper的数据结构

    zk是以key-value的形式存储的，用/分割路径元素的。

    # CAP理论

    CAP理论指对于分布式系统来说，仅能满足其中两点，即要嘛CP，要嘛AP：

    • C（一致性）
    • A（可用性）
    • P（分区容错性），分布式系统在网络分区时发生故障，仍需能够保证对外提供满足一致性、可用性的服务

    zk保证了CP，eruka保证了AP，nacos既可支持CP也可以支持AP。

    # BASE理论

    • 基本可用，分布式故障时，允许损失部分可用性（服务降级）
    • 软引用，允许中间状态
    • 最终一致性

    备注：中间状态指的是不同的data replication（数据备份节点）之间的数据更新出现延迟时的最终一致性*

    BASE理论是对CAP理论中的一致性和可用性进行权衡的一个结果。核心思想：无法做到强一致性，但每个节点应用都应该根据自身的业务特点，达到最后的最终一致性

    # Znode数据结构

    zk中所有存储的数据都是由znode组成，并以key/value形式存储数据的。

    znode的状态属性：

    key 描述
    cZxid 创建节点的事务id
    ctime 创建节点的时间
    mZxid 修改节点的事务id
    mtime 修改节点的时间
    pZid 该节点的子节点列表最后一次修改的事务id；子节点列表变更才会触发，修改子节点内容不触发
    cversion 子节点版本号，子节点修改版本号+1
    dataversion 数据版本号，数据每次修改版本号+1
    aclversion 权限版本号
    ephemeralOwner 创建该临时节点的会话的sessionId
    dataLength 数据长度
    numChildren 子节点数量

    # Session的基本原理

    # Session的创建

    • SessionId：会话id
    • Timeout：会话超时时间
    • TickTime：下次会话超时时间
    • isClosing：标记一个会话是否已经被关闭

    # Session的状态

    1. connecting：连接中
    2. connected：已连接
    3. closed：已关闭

    # 会话超时管理（分桶策略+会话激活）

    zookeeper的leader服务运行期间会对会话超时检查，时间间隔是 ExpirationInterval，单位是毫秒，默认值是 tickTime，每隔 tickTime 进行一次会话超时检查。

    ExpirationTime 的计算方式：

    ExpirationTime = CurrentTime + SessionTimeout;
ExpirationTime = (ExpirationTime / ExpirationInterval + 1) * ExpirationInterval;

    # ACL命令行

    • getAcl：获取某个节点的权限信息
    • setAcl：授权某个节点的权限信息
    • addauth：输入认证授权信息；注册时输入明文密码，存储为加密

    # ACL权限

    ACL权限 ACL缩写 允许的操作
    CREATE c 创建子节点
    DELETE d 删除子节点
    READ r 获取节点数据和子节点
    WRITE w 设置的节点数据
    ADMIN a 设置acl权限

    # 数据同步

    zookeeper实现数据同步主要依赖zab协议，zab协议主要分为两个部分：

    • 消息广播
    • 崩溃恢复

    # 消息广播

    zookeeper使用单一进行leader来接收和处理客户端请求，采用zab原子广播协议，向所有follow广播消息，超过半数的follow回应ack，leader会再次向所有follow发送commit消息，将此次提案提交，这个过程称为2pc事务提交。

    # 崩溃恢复

    当leader服务崩溃或与半数以上follow失去通信，那么就会进入崩溃恢复模式，需要重新选举一个leader服务。

    zab协议恢复模式策略：

    • 选举zxid最大的节点作为leader
    • 新leader将事务日志中未提交的消息进行处理

    # Leader选举原理

    # 选举状态

    • LOOKING：竞选状态
    • FOLLOWING：随从状态。同步leader状态，参与选举
    • OBSERVING：观察状态。同步leader状态，参与不选举
    • LEADING：领导状态

    # 选举重要参数

    1. myid：服务器id
    2. zxid：事务id，越大代表数据越新
    3. epoch-logicalclock：逻辑时钟，同一轮逻辑时钟相同，每投完一轮值增加

    # 选举权重

    1. 优先比较epoch
    2. 检查zxid
    3. zxid相同看myid

    # 分布式锁原理

    # 排他锁

    根据zk的同级节点唯一性原则，c1客户端创建节点/exclusive_lock/lock；其他客户端创建该节点时会失败，这时可以给节点注册一个watch监听事件，当c1释放锁删除节点，watch监听到就可以去创建节点（争抢锁）

    # 共享锁

    1. 客户端创建临时节点
    2. 获取所有子节点列表
    3. 判断是否获得锁，对于读请求，比自己序号小的都是读请求或者没有比自己序号小的，就获得锁；对于写请求，自己不是最小序号，就不能获得到锁
    4. 如果没有获得到共享锁，对于读请求，在向比自己小的最后一个写请求注册监听器；对于写请求，在向比自己小的最后一个请求注册监听器
